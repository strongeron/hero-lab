# Hero Lab — design spec

**Date:** 2026-07-13
**Status:** Accepted & implemented (2026-07-16)
**Author:** Gleb Stroganov

## Resolution

- Repo slug: **`hero-lab`** (descriptive "Hero & landing-page lab" lives in the tagline).
- Hero name: **Dither**. Wild 01/02 collapsed into one hero + Problem/Fix state toggle.
- Internal file renames: **done** (not deferred) — `components/wild/*` → `heroes/dither/*`,
  all `Wild*` identifiers → `Dither*`, for a coherent public repo.
- Phases 1–3 + 5 shipped; engine abstraction (Option C) still deferred to engine #2.

## Problem

The repo is currently a single extracted hero (the "Wild 02" dithering-shader scene +
live control panel). The intent is bigger: a **public, shareable lab for exploring hero
sections** — many heroes, eventually many background *engines* (shader, 3D, gradient) and
layout variants — starting with the one hero we already have, renamed to something a
stranger understands.

Today `main.tsx` hard-mounts one component. That's a hero, not a lab. This spec defines the
smallest architecture that earns the word "lab" without over-building engines we don't have yet.

## Constraints

- **Public portfolio piece.** Strip client branding (ACME header, client copy) into a
  swappable *content preset* — the lab shell ships neutral; the client skin is one optional preset.
- **Start narrow.** Ship with only the current dithering hero (its Problem + Fix states). No
  3D, no gradient engine, no multi-layout yet — but the seams must make those cheap to add.
- **Stay light.** Keep the 4 runtime deps (react, react-dom, @paper-design/shaders-react,
  web-haptics). Don't pull in three.js etc. until a 3D hero actually lands.
- **No client leak.** No client / `wild` / `magicpath` codenames in public-facing names, IDs,
  URLs, or the repo name.

## Naming decisions

**Repo / project:** `hero-shader-playground` → **`hero-lab`** (confirmed direction).

**The hero itself** (currently "Wild 01/02", files `HeroWild01.tsx`): the two variants are one
scene with an `initialState: 'problem' | 'fix'` flag. Collapse into **one named hero with a
Problem⇄Fix state toggle** (the toggle already exists in the panel). Proposed public name:

| Option | id | Read |
|--------|-----|------|
| **Dither** ⭐ | `dither` | Technique-forward, ownable, matches the craft term. The hero *is* the dither. |
| Dot Field | `dot-field` | Evokes the visual; weaker on "what is it." |
| Signal Field | `signal-field` | Observability metaphor (dots = signals, hover reveals failures). On-concept but faintly echoes the origin brand. |

Recommendation: **Dither**. Queryless default = Dither in fix-first (the old "Wild 02");
`/?hero=dither&state=problem` gives the old Wild 01.

## Options considered

**A — Rename only, keep single-mount.** Cheapest. But it stays a one-hero repo; "lab" is a lie.
No path to add heroes without new entry points. Rejected — doesn't meet the goal.

**B — Lean registry + shell (chosen).** A `heroes` registry (array of entries), a lab shell that
renders a hero switcher + the docked control panel, URL routing by `?hero=<id>`. One hero entry
for now (Dither). Adding a hero later = one registry entry + its component. Engine differences
(shader vs 3D) are handled per-entry, not by a premature abstraction layer.

**C — Full engine-interface abstraction.** Formal `HeroEngine` interface, per-engine panel
adapters, layout×background×content matrix up front. Correct eventual shape, but it's
speculative generality with one hero in hand. Rejected for now — build it when the second engine
forces the interface (YAGNI).

## Chosen direction

Option **B**. Minimal registry-driven shell, one hero (Dither), client content demoted to a
preset. The abstraction grows the day a second engine lands — not before.

### Target shape

```
src/
  main.tsx                 mounts <HeroLab/>
  lab/
    HeroLab.tsx            shell: header + hero switcher + docked control panel + URL routing
    registry.ts           heroes = [{ id:'dither', name:'Dither', component, panel, states }]
  heroes/
    dither/               (renamed from components/wild/*)
      DitherHero.tsx        (was HeroWild01.tsx)
      ditherStore.ts        (was wildStore.ts)
      DitherPanel.tsx       (was WildInspectorSection.tsx)
  content/
    presets.ts            copy/nav presets — 'neutral' (default) + client skin (optional)
  playground/ themes/ utils/   (unchanged infra)
```

URL: `/?hero=dither&state=fix` (state optional; defaults to fix).

## Scope

**In:**
1. Rename repo/project/title to `hero-lab`.
2. Rename the hero to **Dither**; collapse Wild 01/02 into one entry + Problem/Fix state.
3. Registry + `HeroLab` shell + hero switcher + `?hero=` routing (one entry today).
4. Strip client content → `presets.ts` with a neutral default; client skin as opt-in preset.
5. README + screenshot refresh for the lab framing.

**Out (non-goals, later phases):**
- 3D hero / react-three-fiber engine.
- Additional shader backgrounds (metaballs, mesh gradient).
- Multiple layout templates (split/centered/left-rail).
- The formal `HeroEngine` interface (Option C) — deferred until engine #2.
- Any deploy/GitHub push (separate step after approval).

## Phases

1. **Rename** — folder, `package.json`, README title, `<title>`, hero id/labels. Low risk.
2. **Registry + shell** — extract `HeroLab` + `registry.ts`; `main.tsx` mounts the shell; Dither
   is entry #1. Wild 01/02 → one entry with a state toggle.
3. **Content presets** — pull client copy out of the hero into `presets.ts`; neutral default.
4. **Internal file renames** *(optional, careful)* — `HeroWild01.tsx`→`DitherHero.tsx` etc. Higher
   churn on large files; can defer if risky.
5. **Polish** — README, screenshots, then (separately) push to GitHub.
